# README + Dashboard — Design Doc

**Date** : 2026-05-08
**Owner** : Etienne Brun
**Status** : Approved (brainstorming validated)

> **Methodological framing** : we build the presentation layer (README + dashboard) BEFORE the headline run. This is a pre-registration discipline: by locking the chart shapes and table columns before seeing the numbers, we prevent post-hoc bias in how we frame which framework "won". The canary results (1 trial × 8 frameworks, already in `results/canary-*.json`) act as preliminary dev data — they are clearly partial, so dashboarding against them carries no risk of cherry-picking.

## Goal

Ship a portfolio-grade presentation layer for the bench results consisting of:
1. A `summarize.py` script that aggregates `results/*.json` into stats consumable by the README and dashboard.
2. A `README.md` at the repo root (primary artifact for hiring managers per design doc v2.4).
3. A Next.js 15 + shadcn/ui + Tremor dashboard in `dashboard/`, deployable on Vercel.

## Non-goals

- Not building per-framework `ANALYSIS.md` files in this phase — those are content, not infrastructure (deferred to post-headline-run polish phase).
- Not adding interactive runs / live model swaps in the dashboard.
- Not building an embed card service (`/embed/{framework}` from earlier design doc) — over-engineering for MVP.
- Not building an automatic CI cron — out of scope for v2.4.

## Components

### Component 1 — `scripts/summarize.py`

Pure Python stdlib (no extra deps). Reads `results/*.json` from a directory (default: `results/`, configurable via `--input` for `results/headline/`). Per framework, computes:

- `count_total`, `count_valid`, `success_rate` (% valid runs)
- Latency: `p50`, `p95`, `mean`, `max` seconds
- Tokens: mean input, mean output, mean total
- Tool calls: mean (with `null` if any framework reports `None` for the metric)
- `hit_step_limit_rate`: % runs that hit MAX_STEPS
- `mean_judge_score`: /20 average from `judgment.relevance + score_coherence + justification_quality + format` (when judge has run)
- `estimated_cost_eur`: `(in_tokens * 2.0 + out_tokens * 12.0) / 1_000_000` (Gemini 2.5 Flash pricing per 1M tokens)

Outputs:
1. `dashboard/data/summary.json` — single JSON with `{"frameworks": [{...stats...}], "metadata": {...}}` (machine-readable)
2. `results/summary.md` — Markdown table for quick CLI inspection

Single command: `python scripts/summarize.py [--input results/]`. Idempotent — re-running on headline data overwrites both outputs.

**Design decision** : separating "machine output" (JSON) and "human output" (MD) lets the dashboard consume one without parsing the other. The README links to `summary.md` for terminal-friendly browsing.

### Component 2 — `README.md`

Sections fixed pre-data:
1. **Hero** (1 paragraph): "Cross-language benchmark of 8 LLM agent frameworks (Python + TypeScript), same model + task + tools, only the framework varies. Goal: pick the right framework for your team's stack."
2. **Leaderboard** : Markdown table with placeholder rows. Auto-updated by `summarize.py` between sentinel comments `<!-- LEADERBOARD-START -->` / `<!-- LEADERBOARD-END -->`.
3. **Charts** : Two PNG images linked from `dashboard/public/charts/` (rendered at dashboard build time via Tremor → Puppeteer screenshot, OR alternative: matplotlib in a small `scripts/render_charts.py`). Decision deferred to implementation; simplest option is a link to the live dashboard.
4. **Frameworks** : list with link to each `/dashboard#{framework}`.
5. **What I learned** : 5 narrative bullets — draftable now from canary observations (e.g. "frameworks with built-in step caps converge faster than the manual loop"; "CrewAI's task-driven framing pushes the model to fewer tool calls but lower-quality picks").
6. **Reproduce** : concrete commands (env var, `uv pip install -e .`, `python harness/run_bench.py --framework all --all-jobs --trials 30`, `python scripts/summarize.py`).
7. **Caveats** : 3 honest limits — (a) Flash vs Pro tradeoff, (b) self-judging bias, (c) per-framework metric availability gaps (e.g. CrewAI tool_calls historically zero before fix).

### Component 3 — `dashboard/`

Stack: Next.js 15 App Router (RSC by default), shadcn/ui (cards, tables, buttons), Tremor (Tailwind-based charts), Vercel deploy.

Routes:

| Route | Server-rendered | Components |
|---|---|---|
| `/` | RSC | Hero card + Leaderboard (sortable, shadcn DataTable) + Pareto chart + Tokens stack chart |
| `/{framework}` | RSC | Framework header + per-framework stats card + sample raw_output + distribution histogram |
| `/methodology` | RSC | Task spec, model, scoring rubric, env, caveats — markdown content rendered via MDX |

Headline charts (2 only — discipline of the v2.4 design doc):
- **Pareto Cost vs Success Rate**: scatter, x = mean cost €, y = success rate %, point per framework, dot size = mean tool_calls. Pareto frontier highlighted.
- **Tokens vs Baseline Stacked Bar**: x = framework, y = mean input + mean output stacked, baseline-python as reference baseline (always 100%, others as % of baseline).

Data flow:
- `summarize.py` → `dashboard/data/summary.json` (committed)
- `lib/stats.ts` exports `Summary`, `FrameworkStats` types matching the JSON shape
- All pages read `data/summary.json` at build time via RSC; no API routes, no client-side fetch

Pre-headline placeholder behavior:
- If a framework has no entries in summary, dashboard shows it with "—" in cells and a `Coming soon` badge.
- If `summary.json` is empty (no runs at all), dashboard shows a "Run pending" page with reproduce instructions.

## Data flow

```
results/canary-*.json (or results/headline/*.json)
        │
        ▼
  summarize.py
        │
        ├──→ dashboard/data/summary.json  (machine, RSC reads at build)
        ├──→ results/summary.md          (human, terminal browsing)
        └──→ README.md leaderboard table (sed-inserted between sentinels)
```

## Error handling

- `summarize.py` skips malformed JSON files with a warning to stderr; exits 0 if at least one file processed.
- Dashboard handles missing fields with `??` fallbacks (TypeScript optional chaining); never crashes on partial data.
- README leaderboard auto-update is non-destructive: only content between sentinels is rewritten; the rest of README is preserved.

## Testing

- `summarize.py`: unit test reading a fixture results dir, asserting computed stats. Add `scripts/test_summarize.py` (~50 LOC, 3-4 fixture cases).
- Dashboard: TypeScript strict mode catches schema mismatches at build time. No additional test framework — visual inspection sufficient for MVP.
- README leaderboard insertion: tested by running `summarize.py` and grepping for the sentinels post-run.

## Scope estimate

- `summarize.py` + tests: 0.25 weekend (~3h)
- `README.md`: 0.25 weekend (~3h, mostly writing)
- Dashboard: 0.5-1 weekend (~6-12h, Next.js + Tremor + shadcn + 2 charts + 3 routes)
- **Total**: ~1-1.5 weekends pre-headline-run

## Sequencing

1. `summarize.py` first (testable on existing canary results)
2. `README.md` (consumes summarize.py output, fillable now)
3. `dashboard/` (consumes summary.json, deployable on Vercel with canary data)
4. **THEN** headline run, then re-run `summarize.py`, dashboard auto-updates
